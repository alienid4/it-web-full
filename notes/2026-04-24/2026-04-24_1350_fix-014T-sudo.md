# 2026-04-24 13:50 — 014T 深度檢查失敗修法 (同 sec9c2 老帳)

**症狀**：對 SECSVR198-014T 跑深度檢查 → `fatal: Missing sudo password`
**根因**：014T 缺少「`sysinfra` 的 passwordless sudo」設定
**影響**：不只深度檢查，**任何對 014T 的 ansible become:true 操作都會掛**（TWGCB 掃描、帳號盤點、Linux 初始化…）

> **備註**：本案已不使用 `ansible_svc` 帳號，全部操作走 `sysinfra`。所以**建立獨立的 sudoers 檔**不要去動既有的 `ansible_svc` 檔（若有）。

## 修法（2 選 1）

### A. 從 13 直接 ssh root 修（最快，推薦）

獨立檔 `/etc/sudoers.d/sysinfra-nopasswd`，避免沿用棄用的 `ansible_svc`：

```bash
ssh root@<014T-ip> "
  echo 'sysinfra ALL=(ALL) NOPASSWD:ALL' > /etc/sudoers.d/sysinfra-nopasswd
  chmod 440 /etc/sudoers.d/sysinfra-nopasswd
  visudo -c
  echo '---'
  echo 'verify:'
  sudo -u sysinfra sudo -n whoami
"
```

預期末段印：
```
verify:
root
```

### B. 人工到 014T 主機修

登入 014T，建立 `/etc/sudoers.d/sysinfra-nopasswd`，內容一行：
```
sysinfra ALL=(ALL) NOPASSWD:ALL
```
存檔後 `chmod 440 /etc/sudoers.d/sysinfra-nopasswd && visudo -c`。

## 順便：可能的其他主機

建議批量檢查**所有受監控主機**是否缺 `sysinfra NOPASSWD`：

```bash
cd /opt/inspection/ansible
sudo -u sysinfra ansible -i inventory/hosts.yml all \
  -m shell -a "sudo -n whoami 2>&1 || echo SUDO_FAIL_on_$(hostname)" \
  --vault-password-file /opt/inspection/.vault_pass 2>&1 | tail -30
```

**任何回 `SUDO_FAIL_...` 的主機**都要套 A 的修法。
